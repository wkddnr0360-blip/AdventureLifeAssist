# 보안 설계

## 관리자 권한

관리자 판정은 다음 중 하나를 만족해야 합니다.

- Firebase ID 토큰의 `admin` custom claim이 `true`
- ID 토큰의 이메일이 `wkddnr0360@naver.com`이고 `email_verified`가 `true`

앱 화면에서 관리자 메뉴를 숨기는 것만으로 끝내지 않고 Firestore Security Rules가 같은 조건을 다시 검사합니다. `server-tools/set-admin-claim.mjs`는 지정 이메일 외의 계정에는 claim을 부여하지 않도록 제한했습니다.

## 관리자 콘솔의 범위

관리자 콘솔은 운영에 필요한 사용자 문서, 공개 프로필·통계, 접속 상태, 알람·로그·메모·계획·업적, 게시물·댓글·신고, 친구 관계, 대화방 메타데이터를 읽어 구조적으로 표시합니다.

다음 정보는 표시하지 않습니다.

- Firebase Authentication 비밀번호: Firebase에서도 원문 비밀번호를 조회할 수 없습니다.
- DM 메시지 본문: 대화 참여자만 읽도록 규칙에서 제한했습니다.
- 서비스 계정 비밀키: 앱·패키지 어디에도 포함하지 않았습니다.

## 개인 데이터와 공개 데이터 분리

- `users/{uid}` 및 하위 로그·알람·메모·계획·업적은 본인과 관리자만 읽습니다.
- `publicProfiles`, `publicStats`, 공개 `presence`에는 랭킹·친구 기능에 필요한 최소 데이터만 게시합니다.
- 사용자가 랭킹 공개를 끄면 공개 통계 숫자를 0으로 정리합니다.
- 사용자가 온라인 공개를 끄면 일반 공개 presence를 제거합니다.
- 운영용 `adminPresence`는 사용자 본인만 쓰고 관리자만 읽습니다.
- 현재 공부 과목은 `shareStudyStatus`가 켜진 경우에만 공개합니다.

## 친구와 DM

- 친구 요청은 보낸 사용자와 받은 사용자만 읽습니다.
- 친구 관계 문서는 관계 당사자만 읽습니다.
- 대화방 메타데이터와 메시지는 `memberIds`에 포함된 사용자만 읽습니다.
- 읽음 표시는 호출자 자신의 UID 필드만 변경할 수 있습니다.
- 일반 사용자는 자신이 포함되지 않은 대화방을 만들거나 구성원을 교체할 수 없습니다.
- 관리자도 기본 규칙상 DM 본문을 우회 열람하지 않습니다.

## 라운지

- 좋아요 배열 변경은 사용자가 자신의 UID만 한 번 추가하거나 제거할 수 있도록 제한했습니다.
- 댓글 삭제는 댓글 작성자, 게시물 작성자, 관리자에게만 허용합니다.
- 신고 문서는 로그인 사용자가 생성하고 관리자만 읽습니다.
- 게시물 작성 UI에서 이메일·전화번호·외부 링크 형태를 차단하고, 규칙에서도 길이와 필수 필드를 검사합니다.

## 추가 권장 사항

- Firebase App Check를 활성화해 승인되지 않은 클라이언트의 남용을 줄입니다.
- Firestore·Realtime Database 사용량 경고와 예산 알림을 설정합니다.
- 관리자 계정에 강한 비밀번호와 다중 인증을 적용합니다.
- Security Rules를 바꿀 때 Emulator Suite의 allow/deny 테스트를 CI에 추가합니다.
- 서비스 계정 JSON, `.env`, 개인 API 토큰은 절대 Git 저장소에 올리지 않습니다.
